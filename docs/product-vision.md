# PeerPedia — Product Vision

**版本:** 2026-06-25
**状态:** 锁定（后续决策以此文档为基准回溯）

---

## 产品定义

PeerPedia 是 **AI 驱动的学术出版**。

不是"加了 AI 功能的学术平台"，不是"P2P 协议加个聊天界面"。
AI 是产品，P2P 是引擎。用户看到的是 Mother，不是 CLI。

---

## 一句话

> 学者用自然语言描述研究 → 论文被发布、被评审、被发现。
> 中间的 git、签名、同步、sedimentation 全是透明的。

---

## 设计原则

这些原则继承自 Steve Jobs 的产品哲学，作为后续所有功能决策的评判标准：

### 1. 最好的界面是没有界面

GUI 元素（菜单、按钮、表单）是认知负担。界面应该只在用户需要做决策时出现。
默认情况下，用户说话，Mother 执行，事情发生。

### 2. 砍掉所有需要"解释"的东西

如果用户需要读文档才能完成一个操作，那就是设计失败了。
注册应该是"你是谁"→ 设密码 → 完成。不是"请理解 Ed25519 密钥派生、存储 salt、签名 commit"。

### 3. 第一次接触决定去留

用户打开 PeerPedia 的前 30 秒，必须感受到"对了"——
Mother 主动问候，告诉用户他关心的论文发生了什么，然后问一句"你想做什么？"

### 4. 技术复杂度是内部事务

git 后端、Ed25519 签名、sedimentation 算法、P2P 同步协议——这些都必须存在，
但用户永远不应该知道它们存在。对外只暴露自然语言和结果。

### 5. 只做一件事，做到让人无法忽视

PeerPedia 的"一件事"：让学者发布论文、获得同行反馈、发现相关研究。
不是"学术社交网络"，不是"文献管理系统"，不是"LaTeX 编辑器"。
如果某个功能不直接服务于这个闭环，它就不应该存在。

## 产品形态

```
                 ┌─────────────────────────┐
                 │        Mother           │  ← 用户唯一可见的界面
                 │   (自然语言对话层)        │
                 └───────────┬─────────────┘
                             │ 意图解析 → 命令调度
                 ┌───────────▼─────────────┐
                 │     CLI 命令层           │  ← 高级用户模式 / 调试入口
                 │  (article, review, …)    │
                 └───────────┬─────────────┘
                             │
        ┌────────────────────┼────────────────────┐
        │                    │                    │
   ┌────▼─────┐      ┌──────▼──────┐      ┌──────▼──────┐
   │  Git 后端 │      │  SQLite DB  │      │  P2P 传输层  │
   │ (签名/同步)│      │ (元数据/索引)│      │ (HTTP 协议)  │
   └──────────┘      └─────────────┘      └─────────────┘
```

- **Mother** 是产品。用户只跟 Mother 交互。
- **CLI** 是引擎室和高级用户调试入口。功能照常开发，但评判标准是"Mother 能不能可靠调用它"。
- **Git + DB + P2P** 是基础设施。用户永远不需要知道它们存在。

## Mother 的发展路线

### Phase 0 — Command Agent（当前优先）

Mother 能理解用户意图，翻译成 CLI 命令序列并执行。

```
用户: "把我那篇量子计算的草稿发出去"
Mother: 找到草稿 → article publish → 确认评分 → 完成
```

不需要理解论文内容，只需要正确调度命令。**这是 MVP 的入口。**

### Phase 1 — Knowledge Crawler

遍历 P2P 网络，沿社交图、引用图、标签图建立本地索引。

### Phase 2 — Structure Recognition

识别论文结构、论证关系、交叉引用，构建知识图谱。

### Phase 3 — Writing Assistant

基于 P2P 网络上的真实文献，做引用推荐、结构建议、新颖性检查。

### Phase 4 — Autonomous Agent

Mother 可以独立写作、评审、发现——她的智能来自 P2P 图，不是模型权重。

---

## 功能评判标准

所有新功能、现有功能，用以下三问来评判：

1. **用户需要为这个功能学什么新概念吗？** 如果需要，砍掉或藏到 Mother 后面。
2. **这个功能能在 Mother 的一句话中被触发吗？** 如果不能，它还不完整。
3. **如果砍掉这个功能，有 90% 的用户会注意到吗？** 如果不会，考虑砍掉。
