## TODO-1: Auto-discovery UX (B6 + B7)

**What:** Trigger `discover_network(depth=1)` automatically after `sync pull` and after bootstrap/login. Users shouldn't need to remember `sync discover`.

**Why:** Without auto-discovery, a new user signs up, logs in, and sees no content. They must manually run `sync discover` — which they won't know exists. For trial readiness, discovery must be automatic.

**Depends on:** B4 (discover_network) implementation.

**Files:** `cli/handlers/bundle.py` (_cmd_sync_pull), `cli/handlers/account.py` (_cmd_bootstrap, _cmd_login)

## TODO-2: Server background peer discovery (B3)

**What:** Spawn a daemon thread in `_cmd_server_start` that calls `merge_peers(seed_url)` for each seed peer, then `push_peer_registration` to announce this server to discovered peers.

**Why:** Without this, servers never discover each other. The peer list stays at the seed list + manually configured URLs. The network doesn't grow organically.

**Depends on:** B1 (POST /api/v1/peers), self-URL resolution.

**Files:** `cli/handlers/server.py`

## TODO-3: Self-URL resolution for NAT peers

**What:** Determine the publicly-reachable URL of the local server when behind NAT/proxy. Options: STUN, UPnP, user-configured `--public-url` flag, DNS.

**Why:** `push_peer_registration(server, own_url)` requires `own_url`. A peer behind NAT cannot produce a URL that other peers can reach. Without solving this, peer registration only works for servers with public static IPs.

**Depends on:** Nothing, but blocks B3 fully working.

**Files:** `cli/handlers/server.py`, `config/params.py`

## TODO-4: Git receive.denyNonFastForwards config

**What:** Add `git config receive.denyNonFastForwards true` to article repos as complementary protection to the runtime `_assert_on_main` guard.

**Why:** The runtime guard catches wrong-branch operations. `denyNonFastForwards` prevents force-push to main, which would rewrite history and corrupt sync state. Defense in depth.

**Depends on:** A (branch protection) implementation.

**Files:** `storage/git_backend.py` (init_article_repo)

## TODO-5: arXiv seed importer

**What:** `peerpedia seed arxiv --query "..." --limit N` — fetch arXiv paper metadata via the arXiv API, create metadata-only article stubs.  Each article:

- **Git repo**: empty (no paper content), single commit with frontmatter containing `arxiv_id` and `doi`
- **Status**: `published` — the paper IS published on arXiv, we're just mirroring the metadata
- **Authors**: created as **stub users** with `public_key=None, salt=""` — unclaimed accounts.  Naming convention: use arXiv author name as-is (P2P allows duplicate names)
- **Maintainer**: the `peerpedia` system account (PLATFORM_EMAIL)
- **Initial scores**: all zero — no PeerPedia reviews yet
- **Dedup**: skip if `article.arxiv_id == entry.arxiv_id` already exists

**Why:** Gives new users something to discover immediately.  Without seed content, the social graph and article discovery have nothing to surface.  arXiv provides 2.5M+ papers with freely-redistributable abstracts — a massive cold-start dataset.

**Author claiming flow (future):**
When the real author joins PeerPedia, they can prove ownership of their arXiv paper by:
1. Identifying their arXiv paper in the PeerPedia system
2. Proving they control the author email (e.g., institution email matching arXiv author affiliation)
3. Or: adding a verification token to their arXiv paper's comments/abstract page
4. On verification, their stub user account gets upgraded (public_key set, salt set, etc.)

**Depends on:** PeerPedia system account existing (PLATFORM_EMAIL), arXiv API client

**Files:**
- `peerpedia_core/transport/arxiv_client.py` — thin HTTP client wrapping `http://export.arxiv.org/api/query`
- `peerpedia_core/commands/articles/seed.py` — `seed_arxiv(db, query, limit)` orchestrator
- `peerpedia_core/cli/handlers/seed.py` — `_cmd_seed` handler
- `peerpedia_core/cli/parser.py` — register `seed arxiv` command
- `peerpedia_core/storage/db/models.py` — Article model: add `arxiv_id` (String, nullable, unique) and `doi` (String, nullable) columns
- `tests/test_seed.py` — verify metadata commit, dedup, zero scores

**arXiv API rate limit:** 1 request/second (burst 4).  `seed arxiv --limit 100` = ~25 API calls.

**Example output:**
```
$ peerpedia seed arxiv --query "attention mechanism" --limit 10
Seeding from arXiv...
  ✓ 1706.03762 "Attention Is All You Need" → 2 stubs (Vaswani, et al.)
  ✓ 1810.04805 "BERT: Pre-training of Deep Bidirectional Transformers" → 1 stub (Devlin)
  ...
Seeded 10 articles, created 23 user stubs.
```
