# PeerPedia — Product Vision

**版本:** 2026-06-25
**状态:** 锁定（后续决策以此文档为基准回溯）

---

## 产品定义

PeerPedia 是 **AI 驱动的学术出版**。

不是"加了 AI 功能的学术平台"，不是"P2P 协议加个聊天界面"。
AI 是产品，P2P 是引擎。用户看到的是 Mother，不是 CLI。

---

## 一句话

> 学者用自然语言描述研究 → 论文被发布、被评审、被发现。
> 中间的 git、签名、同步、sedimentation 全是透明的。

---

## 设计原则

这些原则继承自 Steve Jobs 的产品哲学，作为后续所有功能决策的评判标准：

### 1. 最好的界面是没有界面

GUI 元素（菜单、按钮、表单）是认知负担。界面应该只在用户需要做决策时出现。
默认情况下，用户说话，Mother 执行，事情发生。

### 2. 砍掉所有需要"解释"的东西

如果用户需要读文档才能完成一个操作，那就是设计失败了。
注册应该是"你是谁"→ 设密码 → 完成。不是"请理解 Ed25519 密钥派生、存储 salt、签名 commit"。

### 3. 第一次接触决定去留

用户打开 PeerPedia 的前 30 秒，必须感受到"对了"——
Mother 主动问候，告诉用户他关心的论文发生了什么，然后问一句"你想做什么？"

### 4. 技术复杂度是内部事务

git 后端、Ed25519 签名、sedimentation 算法、P2P 同步协议——这些都必须存在，
但用户永远不应该知道它们存在。对外只暴露自然语言和结果。

### 5. 只做一件事，做到让人无法忽视

PeerPedia 的"一件事"：让学者发布论文、获得同行反馈、发现相关研究。
不是"学术社交网络"，不是"文献管理系统"，不是"LaTeX 编辑器"。
如果某个功能不直接服务于这个闭环，它就不应该存在。

## 产品形态

### 用户看到什么：四个视图，别无他物

整个前端只有 **四个视图**。没有导航栏，没有工具栏，没有侧边栏，没有设置面板。
Mother 决定用户此刻需要哪个视图。视图之间的切换是 Mother 的事，不是用户的事。

```
┌────────────────────────────────────────────────────┐
│                                                    │
│    ┌──────────────┐  ┌──────────┐  ┌──────────┐   │
│    │              │  │          │  │          │   │
│    │   聊天框      │  │  编辑器   │  │  阅读器   │   │
│    │              │  │          │  │          │   │
│    │  + 聊天历史   │  │          │  │          │   │
│    │              │  │          │  │          │   │
│    └──────────────┘  └──────────┘  └──────────┘   │
│                                                    │
│    默认可见          Mother 说         用户说        │
│                     "打开编辑器"       "让我看看      │
│                     时自动跳转         那篇论文"      │
│                                       时自动跳转      │
└────────────────────────────────────────────────────┘
```

**1. 聊天框 + 聊天历史（默认视图）**

用户打开 PeerPedia，看到的就是这个。Mother 说一句，用户回一句。历史可滚动。
这是"主界面"——也是唯一的界面。不需要点任何东西。

Mother 不只是说文字——她会在聊天流中**嵌入卡片**：

```
Mother:  你目前有 3 篇论文在进行中。
         ┌──────────────────────────────────────┐
         │  Quantum Error Correction       草稿 │
         │  ▸ 上次编辑：2 天前                     │
         │  ▸ 2 位合作者                          │
         │  ▸ 自评：未完成                         │
         │         [继续写] [查看详情]              │
         └──────────────────────────────────────┘
         ┌──────────────────────────────────────┐
         │  Tensor Network Methods      沉积池   │
         │  ▸ 3 条评审 · 均分 3.8                │
         │  ▸ 距离出池：1 天                      │
         │         [查看评审] [回复]               │
         └──────────────────────────────────────┘
         需要我做什么吗？
```

卡片类型：
- **论文卡片**：标题、状态、分数、关键指标、操作按钮
- **用户卡片**：姓名、研究领域、出版物数、是否已关注
- **评审卡片**：评分星级、评论摘要、回复状态
- **通知卡片**：事件类型、触发者、关联论文

卡片里的按钮（"继续写"、"查看评审"）被点击后，Mother 执行对应操作——可能是跳转到编辑器/阅读器，也可能直接在聊天流里展开详情。

卡片不是新视图。**卡片就是聊天消息的一种格式。**

**2. 编辑器**

Mother 说"要不要开始写？"→ 用户说好 → 跳转到编辑器。
写完一段，用户说"帮我看这段"→ 内容发回给 Mother。
编辑器不是独立应用——它是聊天的一个视图状态。

**3. 阅读器**

用户说"让我看那篇论文"→ Mother 把论文渲染出来 → 跳转到阅读器。
用户可以在阅读模式下标注、评论、评分。
阅读器也不是独立应用——它只是 Mother 呈现内容的一种方式。

**4. 终端日志（可折叠，默认收起）**

资深用户可以展开一个终端面板，看到 Mother 在后台执行了什么 CLI 命令。
但这**不是第四个视图**——它是一条半透明的底部横条，像浏览器的开发者工具。
默认隐藏。不需要的用户永远不知道它存在。

### 底层架构

```
                 ┌─────────────────────────┐
                 │      前端（4 视图）        │
                 │   聊天框 / 编辑器 / 阅读器   │
                 └───────────┬─────────────┘
                             │ 自然语言 ↔ 意图解析
                 ┌───────────▼─────────────┐
                 │        Mother Agent     │  ← 产品大脑
                 │   (意图 → 命令调度 → 反馈) │
                 └───────────┬─────────────┘
                             │ function calling
                 ┌───────────▼─────────────┐
                 │     CLI 命令层           │  ← 引擎室
                 │  (article, review, …)    │
                 └───────────┬─────────────┘
                             │
        ┌────────────────────┼────────────────────┐
        │                    │                    │
   ┌────▼─────┐      ┌──────▼──────┐      ┌──────▼──────┐
   │  Git 后端 │      │  SQLite DB  │      │  P2P 传输层  │
   │ (签名/同步)│      │ (元数据/索引)│      │ (HTTP 协议)  │
   └──────────┘      └─────────────┘      └─────────────┘
```

- **四个视图** 是用户看到的一切。没有"功能菜单"。
- **Mother Agent** 是大脑。意图解析、命令编排、视图切换——全部由她决策。
- **CLI** 是引擎室。用户可以选择看到它（展开终端日志），但不是必须的。
- **Git + DB + P2P** 是基础设施。用户永远不需要知道它们存在。

## Mother 的发展路线

### Phase 0 — Command Agent（当前优先）

Mother 能理解用户意图，翻译成 CLI 命令序列并执行。

```
用户: "把我那篇量子计算的草稿发出去"
Mother: 找到草稿 → article publish → 确认评分 → 完成
```

不需要理解论文内容，只需要正确调度命令。**这是 MVP 的入口。**

### Phase 1 — Knowledge Crawler

遍历 P2P 网络，沿社交图、引用图、标签图建立本地索引。

### Phase 2 — Structure Recognition

识别论文结构、论证关系、交叉引用，构建知识图谱。

### Phase 3 — Writing Assistant

基于 P2P 网络上的真实文献，做引用推荐、结构建议、新颖性检查。

### Phase 4 — Autonomous Agent

Mother 可以独立写作、评审、发现——她的智能来自 P2P 图，不是模型权重。

---

## 功能评判标准

所有新功能、现有功能，用以下三问来评判：

1. **用户需要为这个功能学什么新概念吗？** 如果需要，砍掉或藏到 Mother 后面。
2. **这个功能能在 Mother 的一句话中被触发吗？** 如果不能，它还不完整。
3. **如果砍掉这个功能，有 90% 的用户会注意到吗？** 如果不会，考虑砍掉。
